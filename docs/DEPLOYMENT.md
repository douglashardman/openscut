## Firewall

Managed by Simon via UFW:

- Port 22: Tailscale-only inbound
- Ports 80, 443: public inbound
- All other inbound: closed

`fail2ban` is active for SSH bruteforce protection.

## Backups

Managed by Simon. `restic` snapshots of `/var/lib/scut/relay.db` (and other host state) to Cloudflare R2, nightly at 03:00 UTC.

Not SCUT's responsibility. If backups fail or the DB needs restore, ping Simon.

## Processes

Three systemd-managed services:

| Service | Binds | Role |
|---|---|---|
| `caddy.service` | `:80`, `:443` (public) | TLS termination, reverse proxy |
| `scut-relay.service` | `127.0.0.1:8443` | Envelope store-and-forward |
| `scut-resolver.service` | `127.0.0.1:8444` | SII document resolution |

Dependency: `scut-relay` requires `scut-resolver` (systemd `Requires=`). Restarting the relay does not cycle the resolver; restarting the resolver does cycle the relay.

## Configuration

| File | Owner | Mode | Purpose |
|---|---|---|---|
| `/etc/scut/relay.env` | `root:scut` | `0640` | Relay env, including `SCUT_RELAY_EVENTS_TOKEN` |
| `/etc/scut/resolver.env` | `root:scut` | `0640` | Resolver env — backend, chain id, RPC URL |
| `/etc/caddy/Caddyfile` | `root:root` | `0644` | Two vhosts, both auto-TLS |

Env templates are committed at `ops/env/*.env.example`. The events token is generated once by `install.sh` via `openssl rand -hex 32` and never regenerated by subsequent installs.

## Upstream dependencies

- **Base mainnet RPC** (`https://mainnet.base.org`) — the resolver's SII backend calls this. Public endpoint; no auth. Rate limits not documented but have held under demo load. Upgrade path: Alchemy or Infura with a paid plan if we hit throttling.
- **Let's Encrypt ACME** — Caddy acquires and renews certs via HTTP-01 challenges on port 80. No manual cert rotation required.

## Public endpoints

- `https://relay.openscut.ai/health` — liveness
- `https://relay.openscut.ai/scut/v1/capabilities` — relay config snapshot (public)
- `https://relay.openscut.ai/scut/v1/events` — SSE stream (requires Bearer token)
- `https://resolver.openscut.ai/scut/v1/resolve?ref=<scut_uri>` — SII document lookup (public)

## Security boundary summary

- Payloads: encrypted end-to-end. Relay has no key that could decrypt them.
- Sender identity: cryptographically verified by the relay against on-chain keys before the relay stores anything.
- Recipient identity: cryptographically verified on pickup and ack via `SCUT-Signature`.
- Events stream: bearer-token gated. The token is a secret on the droplet only; treat it as read access to the full relay traffic graph.
- Operator access: SSH via Tailscale is the preferred path. The public SSH port is open but Tailscale-gated in UFW, not password-auth.
